1-Access to data (documents and structured data) within the Kiewit medallion storage systems is controlled by a separate data engineering group within Kiewit.  Post replication from source we need to make key operational data discoverable (model quantities, component claiming, scheduling, etc. (not social security numbers).  Usage of iceberg tables, like how Palantir stores data, makes data not only in the data lake but also in our data warehouse (Snowflake).  This makes data more discoverable.
2-Broad availability of data: Design model data for most design build projects is not ingested or normalized, scheduling relationship data from InEight is not available, custom-built systems (LAND, KADE, ADAPT, etc.) need to integrate into KIP.  We may also have to partner with Palantir or others to get software applications that ingest key data sources like Source360.
3-Orchestration is a critical capability that will allow us to connect pipelines together & monitor the health of data movements.  Each arrow is a large volume of code that requires care and feeding.  Code is owned by organizations.  Data science is dependent on open-source tools like Airflow and Dagster and we need more holistic software engineering in our data engineers tool belt.  We need to enable reverse engineering of datasets to stop redundancy of dataset creation by data science.  We also need to stop enterprise redundancy with currently 36 Snowflake databases plus an on-premises ODS with many more.
4-We need to not only augment with external companies like phData or InCycle but also build our internal data engineering skillsets up in a single organization with joint priorities shared by data science and data engineering.  Not all data engineers in one organization but key domains only.
