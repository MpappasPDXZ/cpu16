SECTION 3: InEight ARCHITECTURE & AI INTEGRATION
--------------------------------------------------------------------------------

Current Architecture:
- Front End: C#.NET (legacy) -> React (modern, converting)
- Backend: SQL Server (relational database)
- Data Storage: Structured, planned retrieval model

Required Migration:
- Target: Data Lake House Architecture
- Rationale: Support dynamic, unplanned AI queries
- Benefits: Distributed processing, schema flexibility, cost efficiency

Anthropic Skills Framework:
Agent Skills are organized folders containing:
1. SKILL.md file with YAML frontmatter (name, description)
2. Progressive disclosure structure:
   - Level 1: Metadata (~100 tokens) - loaded at startup
   - Level 2: SKILL.md body (<5k tokens) - loaded when skill triggers
   - Level 3: Bundled files (unlimited) - loaded as-needed
3. Executable scripts for deterministic operations
4. Reference documentation for complex workflows

Skills Framework Challenges with SQL Server:
- SQL Server optimized for planned, structured queries
- Anthropic Skills generate dynamic dataset calls
- Unplanned retrievals inefficient on relational architecture
- RAG-based search works with planned storage but limited

Lake House Advantages:
- Supports distributed, unplanned data retrievals
- Schema evolution without breaking downstream consumers
- Time travel for reproducibility
- ACID transactions for data integrity
- Zero-copy architecture across compute engines

Source: Anthropic Engineering Blog, "Equipping agents for the real world with Agent Skills" (October 2025)

SECTION 5: BUILD LOCATION STRATEGY
--------------------------------------------------------------------------------

Architecture Flow:
InEight Front End (React UI) -> InEight Backend (SQL Server) -> KIP Lake (Data Integration) -> AI Agent Applications (Intelligence)

KIP Structure (Four-Layer Platform):

L0: Data Sources
- Enterprise systems (SAP, InEight, GEP)
- P6 schedules, estimate files, 3rd party contracts
- Engineering models, procurement data (Source360)
- Spatial scans and video capture (FieldAI)
- SharePoint, real-time operational streams

L1: Data Factory
- Open-source orchestration (Dagster/Airflow)
- Modern lakehouse architecture (Iceberg tables, Polaris catalog)
- Ingests enterprise data from SAP, InEight, Primavera to data lake
- Model Context Protocol (MCP) servers provide LLMs secure access
- Data lineage tracking

L2: Kiewit Intelligence
- AI-powered data ingestion to open-source tables via Polaris catalog
- 20,000+ predictive algorithms (FastAPI/Azure)
- AI agents for conversational search and document generation
- Centralized data science team drives continuous quality improvement

L3: Automations
- Operational workflow automation (Takeoffs, Estimating, Scheduling, Work Planning)
- Powered by 16-node NVIDIA B200 high-performance compute cluster
- Data science team ramping to deliver operational staff improvements

Decision Framework:
Option A: Build AI applications on Kiewit KIP lake
- Focus: Data integration
- Investment: KIP platform enhancement
- Benefit: Reduced vendor dependency, increased flexibility
- Risk: Requires InEight data extraction and normalization

Option B: Move InEight data into Kiewit infrastructure
- Focus: Platform modification
- Investment: InEight platform changes
- Benefit: Direct integration, potentially faster
- Risk: Vendor dependency, platform lock-in

Recommendation: KIP lake approach
- Leverages existing KIP investment
- Reduces vendor dependency
- Enables multi-source data integration
- Requires ROI analysis and customer demand assessment


SECTION 6: ROI REALITY CHECK & ORDER OF OPERATIONS
--------------------------------------------------------------------------------

Reality Check:
Not all AI investments yield positive ROI. Common failure modes:
1. Unperformant architectures slow systems
2. Degraded user experience from poor integration
3. Expensive token consumption without value delivery
4. Building on unstable foundations creates rework

Prerequisites for Success:
- Lake house migration (SQL Server insufficient for dynamic AI)
- Data quality and availability
- Clear use case definition
- Proper sequencing

Critical: ROI Scoping & Sequencing

ROI Scoping Process:
1. Clearly define the use case
2. Scope effort, timeline, and outcomes
3. Estimate value and ROI
4. THEN evaluate whether investment is justified

Order of Operations (Dependency Chain):
Design Model -> Quantities -> Estimates/Procurement -> Schedules -> Work Plans

Four-Tier Dependency Structure:

Tier 1 (Foundation): 6 use cases
- Design model ingestion
- Takeoff automation
- Quantity normalization
- Visualization standards
- Timeline: 2026
- Cost: ~$3M

Tier 2 (Estimates): 4 use cases
- Estimate creation
- Estimate conformance
- Construction/Engineering/Procurement estimates
- Timeline: 2027 (after Tier 1 proven)
- Cost: ~$4M

Tier 3 (Schedules & Procurement): 5 use cases
- Schedule creation
- Risk analysis
- Vendor selection
- Timeline: 2027-2028 (after Tier 1+2 proven)
- Cost: ~$5M

Tier 4 (Execution): 3 use cases
- Work package creation
- Short interval planning
- Schedule auto-progress
- Timeline: 2028+ (after Tier 1+2+3 proven)
- Cost: ~$3M

Rework Multiplier Example:
If Tier 1 requires $3M and 6 months, but we build Tiers 2-4 in parallel ($12M, 18 months), then discover Tier 1 was wrong:
- Cost to fix Tier 1: $3M
- Cost to rework Tiers 2-4: $12M+
- Total cost: $15M+ (vs. $10M if sequenced properly)

Key Principle: Discipline beats dollars
A well-sequenced $10M engagement delivers more value than a poorly-sequenced $30M engagement.
